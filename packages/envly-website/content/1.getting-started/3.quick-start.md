This guide walks you through the basic workflow: creating a vault, adding secrets, setting up a project, and activating an environment.

## 1. Create Your First Vault

When you launch Envly for the first time, you'll be guided to create a vault. A vault is an encrypted file that stores all your secrets, projects, and environment configurations.

You'll need to choose:
- **Vault name** — a label for this vault (e.g. "Work", "Personal")
- **File location** — where to save the encrypted vault file on disk
- **Authentication method**:
  - **Passphrase** — you enter a master password each time you open the vault
  - **OS Keychain** — the encryption key is stored in your system keychain (macOS Keychain, Windows Credential Manager, or Linux Secret Service) and the vault unlocks automatically

::tip
You can create multiple vaults later for different contexts and switch between them at any time.
::

## 2. Add Secrets

Navigate to the **Secrets** page to start adding your sensitive values. Each secret has:

- **Key** — a unique identifier (e.g. `POSTGRES_PASSWORD_STAGING`)
- **Value** — the secret data (encrypted at rest, masked in the UI)
- **Description** (optional) — a note about what this secret is for
- **Tags** (optional) — labels for filtering (e.g. `database`, `aws`, `staging`)
- **Expiration date** (optional) — Envly will warn you when a secret is about to expire

You can also **bulk import** from an existing `.env` file if you already have secrets in that format.

## 3. Create a Project

Go to the **Projects** page and create a project. A project maps to a local repository on your machine:

- **Name** — a label for the project (e.g. "API Backend")
- **Folder path** — the local path to your project directory
- **Env filename** — the file to create in your project (defaults to `.env`, but you can set `.env.local`, `.env.development`, etc.)

## 4. Add an Environment

Inside your project, create one or more environments (e.g. "development", "staging", "production"). Each environment has its own set of **mappings** that bind local `.env` keys to your global secrets.

For example, in a "development" environment:

| Local Key | Secret |
|-----------|--------|
| `DATABASE_URL` | `POSTGRES_URL_DEV` |
| `API_KEY` | `STRIPE_TEST_KEY` |
| `REDIS_HOST` | `REDIS_LOCAL` |

## 5. Activate the Environment

Click the **activate** toggle on the environment you want to use. Envly will:

1. Resolve all mappings to their current secret values
2. Write the key-value pairs to a secure temporary file (readable only by your user)
3. Create a symlink from your project directory (e.g. `~/code/api/.env`) pointing to the temp file

Your dev server reads the `.env` file through the symlink as usual — no changes to your workflow needed.

::callout{icon="i-lucide-shield-check"}
Only one environment per project can be active at a time. Activating a new one automatically deactivates the previous one.
::

## 6. Lock When Done

When you're finished working, lock your vault (or just close Envly). All temporary files and symlinks are cleaned up automatically, and the decrypted secrets are wiped from memory.
