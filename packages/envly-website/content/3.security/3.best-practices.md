Envly is designed to protect your secrets, but security is a shared responsibility. Follow these recommendations to get the most out of Envly's security model.

## Enable Full-Disk Encryption

Envly encrypts your vault file, but temporary files (containing decrypted secrets while an environment is active) live on disk. Full-disk encryption ensures these are protected even if your machine is lost or stolen.

| Platform | Solution |
|----------|----------|
| macOS | FileVault |
| Windows | BitLocker |
| Linux | LUKS |

## Lock Your Screen

Envly cannot protect against physical access to an unlocked session. Lock your screen when stepping away from your machine.

## Add `.env` to `.gitignore`

Always add your env filename (e.g. `.env`, `.env.local`) to your project's `.gitignore`. While committed symlinks won't leak secrets (they're just broken links on other machines), it's best practice to exclude them entirely.

Most frameworks include `.env` in their default `.gitignore` templates, but it's worth double-checking.

## Use Strong, Unique Secret Values

Since Envly manages your secrets, you never have to remember them. Take advantage of this by using long, randomly generated values for passwords, API keys, and tokens.

## Set Expiration Dates

For secrets that should be rotated regularly (database passwords, API keys with rotation policies), set expiration dates. Envly will warn you as deadlines approach, helping you stay on top of secret rotation.

## Back Up Your Vault

Periodically export a backup of your vault using the export feature in Settings. The export creates an encrypted copy with a passphrase you choose, which you can store securely (e.g. on an encrypted external drive or secure cloud storage).

## Review Active Environments

Before committing code, review which environments are active. While Envly's symlink approach prevents accidental secret leaks, defense in depth is always a good strategy.

## Use Keychain Mode on Trusted Machines

On your personal development machine, keychain mode provides a seamless experience — the vault unlocks automatically with your OS session. Use passphrase mode on shared or less trusted machines where you want an explicit authentication step.

## One Vault Per Context

Consider using separate vaults for different contexts (e.g. work vs. personal, or per client). This limits the blast radius if a vault's passphrase is ever compromised, and makes it easier to share a specific vault with a colleague via the export feature.
