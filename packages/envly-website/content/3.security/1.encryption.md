Envly encrypts all vault data using modern, well-audited cryptographic primitives. This page explains what happens under the hood without requiring deep cryptography knowledge.

## Encryption at Rest

Everything in a vault — secrets, project names, environment configs, mappings — is serialized to JSON and encrypted as a **single blob** using **XChaCha20-Poly1305**, an authenticated encryption scheme (AEAD). Nothing is stored in plaintext on disk.

The vault file on disk looks like this:

```json
{
  "cipher_version": 1,
  "cipher_kind": "passphrase",
  "payload": {
    "ciphertext": "<base64-encoded encrypted data>",
    "nonce": "<base64-encoded random nonce>",
    "salt": "<base64-encoded salt (passphrase mode only)>"
  }
}
```

Only the cipher metadata (`cipher_version`, `cipher_kind`) is unencrypted — everything else is inside the encrypted payload.

## Cipher Modes

### Passphrase Mode

Your master password is stretched into a 256-bit encryption key using **Argon2id**, a memory-hard key derivation function designed to resist brute-force attacks:

| Parameter | Value |
|-----------|-------|
| Memory | 64 MiB |
| Iterations | 3 |
| Parallelism | 4 |
| Output | 256-bit key |

The derived key is then used with XChaCha20-Poly1305 to encrypt and decrypt the vault. Your password is never stored anywhere.

### Keychain Mode

A random 256-bit encryption key is generated and stored in your operating system's credential manager:

| Platform | Storage |
|----------|---------|
| macOS | Keychain |
| Windows | Credential Manager |
| Linux | Secret Service (GNOME Keyring, KWallet, etc.) |

The same XChaCha20-Poly1305 cipher is used for encryption. The vault unlocks automatically when your OS session is authenticated — no password to enter.

## Encryption in Memory

While the vault is unlocked, decrypted data lives in memory. Envly takes several precautions:

- **Zeroizing memory** — all secret values and encryption keys use `Zeroizing<String>`, a Rust type that overwrites memory with zeros when the value is dropped. This ensures secrets don't linger in freed memory.
- **On lock/exit** — the entire in-memory vault is dropped, triggering zeroization of all sensitive data.
- **No logging** — secret-holding types don't implement `Debug` or `Display`, preventing accidental logging of sensitive values.

## Atomic Writes

Vault saves use a safe strategy to prevent corruption:

1. Serialize the vault to JSON
2. Encrypt the JSON
3. Write to a **temporary file**
4. Rename the temp file over the old vault (atomic operation)

Before each write, the previous vault file is copied to a `.bak` backup. If the write succeeds, the backup is removed. If the app crashes mid-write, the backup ensures you don't lose data.

## Cipher Versioning

The vault file header includes a `cipher_version` field. This allows future Envly versions to upgrade the encryption scheme while still being able to decrypt older vaults and re-encrypt them transparently.
