Envly uses a symlink-based approach to inject environment variables into your projects. This design is intentionally safer than writing plaintext `.env` files directly.

## How It Works

When you activate an environment, Envly:

1. **Resolves** all mappings to their current secret values
2. **Writes** the key-value pairs to a secure temporary file in an app-owned directory
3. **Creates a symlink** from your project directory (e.g. `~/code/api/.env`) pointing to the temp file

Your development tools follow the symlink and read the `.env` file as usual.

```
~/code/api/.env  →  ~/.local/share/com.envly.envly/tmp/env-<uuid>
```

## Why Symlinks Are Safer

If you accidentally run `git add .` and commit the symlink:
- On any other machine, the symlink points to a **nonexistent path**
- No secret data is leaked — it's just a broken link
- Compare this to committing an actual `.env` file, which would expose all your secrets in the git history

## Temp File Security

Temporary files are stored in Envly's app data directory with strict permissions:

| Aspect | Detail |
|--------|--------|
| **Directory** | `~/.local/share/com.envly.envly/tmp/` (Linux), `~/Library/Application Support/com.envly.envly/tmp/` (macOS), `%APPDATA%\com.envly.envly\tmp\` (Windows) |
| **Directory permissions** | `0700` (owner only) |
| **File permissions** | `0600` (owner read/write only) |
| **File names** | Random UUIDs (`env-<uuid>`) to prevent guessing |

## Cleanup

Temporary files and symlinks are cleaned up in several scenarios:

- **Deactivating an environment** — the temp file and symlink are removed immediately
- **Locking the vault** — all active environments are deactivated and cleaned up
- **Closing the app** — same as locking
- **On startup** — Envly scans for stale entries from previous sessions that may have crashed. Entries from dead processes or older than 24 hours are automatically cleaned up.

Envly tracks all active temp files in a manifest (`active_envs.json`) so it can always find and clean up its own files.

## Existing File Handling

When creating a symlink, Envly checks what already exists at the target path:

| What's There | What Envly Does |
|-------------|----------------|
| Nothing | Creates the symlink |
| An Envly-managed symlink | Replaces it |
| A regular `.env` file | Backs it up to `.env.bak`, then creates the symlink |
| A symlink to somewhere else | Refuses to overwrite — you need to resolve this manually |

## Recommendations

- **Add `.env` to your `.gitignore`** in all projects. Most frameworks already do this in their default templates.
- **Enable full-disk encryption** (FileVault, BitLocker, LUKS) to protect temp files if your machine is lost or stolen.
- **Review active environments** before committing code — defense in depth matters.
