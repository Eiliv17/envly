Each project can have multiple environments (e.g. development, staging, production). An environment defines a set of **mappings** that bind local `.env` keys to your global secrets.

## How It Works

An environment mapping looks like this:

| Local Key | Points To | Notes |
|-----------|-----------|-------|
| `DATABASE_URL` | Secret: `POSTGRES_URL_DEV` | Development database |
| `API_KEY` | Secret: `STRIPE_TEST_KEY` | Stripe test mode |
| `REDIS_HOST` | Secret: `REDIS_LOCAL` | Local Redis instance |

When you activate the environment, Envly resolves each mapping to the current value of the referenced secret, writes them as `KEY=VALUE` pairs to a secure temporary file, and creates a symlink in your project directory.

## Creating Environments

Inside a project's detail page, create a new environment with a name and optional description. Then add mappings one by one:

1. **Local key** — the variable name your application expects (e.g. `DATABASE_URL`)
2. **Secret** — select which global secret provides the value
3. **Notes** (optional) — explain why this mapping exists or how it differs from other environments

Local keys must be unique within an environment — you can't have two mappings for the same key.

## Activation

Only **one environment per project** can be active at a time, since they all write to the same env file in the project directory. When you activate an environment:

1. Any previously active environment in the same project is automatically deactivated
2. Mappings are resolved to their current secret values
3. A temporary file is written with `0600` permissions (readable only by your user)
4. A symlink is created from your project's env file to the temp file

Your dev server reads the symlink as a regular file — no changes to your workflow needed.

## Deactivation

Deactivating an environment removes the symlink and deletes the temporary file. No trace of the secret values remains in your project directory.

Environments are also automatically deactivated when:
- You lock the vault
- You close the app
- You switch to a different vault

## Live Updates

If you update a secret's value while an environment using it is active, Envly automatically refreshes the symlink. The temp file is rewritten with the new values so your running dev server picks up the change.

## Cloning Environments

You can clone an environment to create a copy of all its mappings. This works both within the same project and across different projects — useful when you want a similar setup with slightly different secrets.

## File Collision Handling

When activating an environment, the target file (e.g. `.env`) might already exist in your project directory. Envly handles this gracefully:

| Situation | What Envly Does |
|-----------|----------------|
| No file exists | Creates the symlink normally |
| An Envly-managed symlink exists | Replaces it with the new one |
| A regular file exists | Backs it up to `.env.bak` and creates the symlink |
| An unknown symlink exists | Refuses to overwrite and asks you to resolve it manually |
